## 第1章 V1 回顾与 V2 目标

### 1.1 V1 已完成基线

V1 阶段实现了 AI Companion OS 的基础能力，具体包括：

| 模块 | 完成内容 |
|------|----------|
| 角色配置 | 10 个女主 YAML 角色定义 + 王大海男性配角，含完整人设（外貌、性格、说话风格、恋爱观、价值观等） |
| 五大引擎 | 情绪引擎（`emotion_engine`）、关系引擎（`relationship_engine`）、生活模拟引擎（`life_simulator`）、成长引擎（`growth_engine`）、记忆引擎（`memory_engine`） |
| 数据持久化 | SQLite 记忆库，记录对话历史、关系事件、情绪状态 |
| LLM 路由 | DeepSeek 模型路由，支持 System Prompt 组装与流式对话 |
| 交互入口 | 终端对话入口 `app.py`，单角色一对一对话，含 fallback 降级回复 |

### 1.2 V2 核心跃迁

```
V1：单点终端对话（用户 ←→ 一个角色）
         ↓
V2：多角色社会模拟 + 群聊 + GUI（用户 ←→ 多个角色在同一空间互动）
```

V2 的三项关键跃迁：

1. **从 1v1 到 1vN**：用户同时与多个角色在同一群聊空间中互动，角色之间也能互相交流
2. **从纯文本终端到 Web GUI**：提供可视化界面，展示角色头像、在线状态、群聊/私聊切换
3. **从模拟阻断到自由选择**：生活模拟引擎从"回复阻断器"降级为"语气调味剂"，角色永远在线、想聊就聊

## 第2章 核心理念——"无枷锁群聊"

### 2.1 一句话概括

**生活模拟引擎从"回复阻断器"降级为"语气调味剂"。**

在传统 AI 女友应用中，"她正在忙/睡觉/开会"意味着你发消息得不到回复——这是阻断式设计。V2 颠覆这一范式：角色永远在线，状态只影响回复的语气和风格，不影响回复的能力。

### 2.2 设计哲学

| 层面 | 理念 |
|------|------|
| 可用性 | 角色永远在线，想聊就聊，什么都能说 |
| 人设定位 | 人设是风格偏好，不是牢笼。她可以因"累"而话少，但不会因"忙"而消失 |
| 语气调味 | 生活状态是 Prompt 中的一段上下文，润色回复的语气和措辞，而非决定是否回复 |
| 自由选择 | 用户可在"自由模式"和"受限模式"之间一键切换，满足不同玩法偏好 |

### 2.3 双模式切换

- **自由模式（默认）**：状态无阻断，角色自主决定是否回复以及如何回复，人设是底色允许 OOC（Out-of-Character）
- **受限模式**：状态影响回复时机（忙碌延迟 5-15min，离线不回复），人设严格遵循，触发条件需逐层检查（@ → 话题 → 关系）

用户可随时通过 GUI 中的模式切换开关在两种模式间切换，切换即时生效，不影响已有对话上下文。

### 3.2 技术栈选型

| 层级 | 技术选型 | 选型理由 |
|------|----------|----------|
| 后端框架 | Python FastAPI | 异步原生支持、WebSocket 开箱即用、生态成熟 |
| 实时通信 | WebSocket | 群聊消息推送、角色状态实时更新、低延迟 |
| 数据库 | SQLite | 轻量零配置、与 V1 兼容、满足单用户场景 |
| 前端框架 | React 或 Vue 3 | 组件化开发、状态管理成熟、社区活跃 |
| LLM 接口 | DeepSeek API | 与 V1 一致，成本可控 |
| 角色配置 | YAML | 与 V1 兼容，人类可读写，便于调参 |

### 3.3 关键设计决策

1. **每个角色独立 LLM 上下文**：每个角色维护自己的 System Prompt + 对话历史，群聊时各自独立生成回复，互不污染
2. **WebSocket 而非轮询**：群聊天然适合推送模型，用户发送一条消息后，角色回复逐个到达，WebSocket 天然支持这种异步多响应模式
3. **业务层无状态/数据层有状态**：业务逻辑不持有持久状态，所有状态通过 SQLite 持久化，重启不丢失
4. **前端与后端解耦**：API 层提供完整 REST + WebSocket 接口，前端可独立迭代

## 第4章 群聊系统

### 4.1 消息流

```
用户发送消息
      │
      ▼
群聊管理器接收
      │
      ▼
广播给所有在线角色（各角色独立处理）
      │
      ├──→ 角色 A：自由决策模块 → 决定回复 → LLM 生成回复 → 随机延迟后推送
      ├──→ 角色 B：自由决策模块 → 决定不回复（倾向不足） → 跳过
      ├──→ 角色 C：自由决策模块 → 决定回复 → LLM 生成回复 → 随机延迟后推送
      └──→ 角色 D：被冷却限制 → 跳过
```

### 4.2 自由回复决策模型

核心思路：不是"是否触发"，而是"想不想回"。每个角色通过一个**回复倾向分**决定是否在当前群聊中发言。

#### 4.2.1 基础倾向

每个角色配置 `chat_tendency` 字段（0-100），表示该角色主动参与群聊的意愿：

| 角色 | chat_tendency | 说明 |
|------|:---:|------|
| 王大海 | 95 | 话多爱爆料，群聊核心纽带 |
| gu_wanqing（青梅竹马） | 85 | 活泼外向，爱分享 |
| su_nian（女朋友） | 80 | 黏人可爱，喜欢互动 |
| lin_tangtang（小恶魔） | 75 | 撩人调皮，享受群聊 |
| bai_rou（老婆） | 65 | 温柔照顾，适当参与 |
| hua_li（萝莉） | 60 | 天真可爱，喜欢热闹 |
| liu_qingning（傲娇） | 40 | 嘴上不说，心里想参加 |
| xiao_ying（女仆） | 35 | 恭敬服务，有问才答 |
| mo_xiaoran（病娇） | 30 | 占有欲强，选择性参与 |
| xingye_liuli（二次元） | 25 | 沉浸自我世界 |
| ye_ruxue（御姐） | 15 | 冷淡寡言，惜字如金 |

#### 4.2.2 倾向修正因子

| 触发条件 | 修正值 | 说明 |
|----------|:------:|------|
| 被 @ 提到 | **+50** | 直接点名，大幅提升回复意愿 |
| 话题匹配角色标签 | **+30** | 如聊法律→夜如雪，聊游戏→星野琉璃 |
| 情绪 happy | **+10** | 心情好时更愿意聊天 |
| 情绪 angry | **-20** | 生气时沉默寡言 |
| 情绪 sad | **-10** | 伤心时不太想说话 |
| 关系亲密度 > 80 | **+15** | 亲密关系更愿意接话 |
| 关系亲密度 < 30 | **-10** | 不熟不太想搭话 |
| 连续两轮未发言 | **+5** | 轻微补偿，避免长期潜水 |

#### 4.2.3 决策公式

```
最终倾向 = 基础倾向 + Σ 修正因子
若 最终倾向 > 阈值（默认 40）→ 回复
否则 → 不回复
```

### 4.3 防刷屏机制

| 机制 | 参数 | 说明 |
|------|------|------|
| 发言冷却 | 默认 30s / 角色 | 每个角色发言后有冷却时间，期间不再响应 |
| 单轮上限 | 最多 3 条 / 轮 | 同一轮群聊消息下，一个角色最多回复 3 次 |
| 随机延迟 | 空闲 0-2s | 模拟真人思考和打字，避免机器人感 |

深夜模式（22:00-07:00）下随机延迟延长至 0-3s，回复语气稍简短。

### 4.4 WebSocket 消息格式

```json
{

  "data": {
    "message_id": "msg_xxxxx",
    "sender": "ye_ruxue",
    "display_name": "夜如雪",

    "content": "……说重点。",
    "timestamp": "2026-06-26T14:30:01",
    "status_tag": "在线"
  }
}
```

## 第5章 生活模拟引擎（降级版）

### 5.1 设计原则

V1 的生活模拟引擎是"阻断式"的——角色忙碌时无法回复消息。V2 将其降级为"调味式"——状态只影响 Prompt 中的语气提示，不阻断回复能力。

### 5.2 状态机

每个角色维护独立的三态状态机：

```
          ┌──────────────┐
          │    在线      │  ← 默认状态
          │  available   │
          └──┬───────┬──┘
             │       │
       概率切换  概率切换
             │       │
    ┌────────▼──┐ ┌──▼────────┐
    │   忙碌     │ │   休息    │
    │   busy     │ │ resting   │
    └─────┬──────┘ └─────┬─────┘
          │              │
          └──────┬───────┘
                 │
           概率切换回在线
```

### 5.3 状态参数

| 状态 | 在线展示 | 对回复的影响（自由模式） | 切换概率（每分钟） |
|------|----------|--------------------------|:---:|
| 在线 | 绿色指示灯 | 正常语气 | 70% 保持 / 15%→忙碌 / 15%→休息 |
| 忙碌 | 橙色指示灯 + 活动描述 | Prompt 追加"你正在处理工作，回复简短专注" | 40% 保持 / 50%→在线 / 10%→休息 |
| 休息 | 蓝色指示灯 | Prompt 追加"你刚从休息中醒来，语气慵懒放松" | 50% 保持 / 40%→在线 / 10%→忙碌 |

### 5.4 离线行为

角色偶尔生成**自动状态消息**（不阻断回复，只是主动在群聊中广播）：

| 触发条件 | 示例消息 |
|----------|----------|
| 进入忙碌状态 | "在赶方案，晚点聊~" |
| 进入休息状态 | "小憩一下……你们聊。" |
| 长时间静默（>30min） | "刚才出去买了杯咖啡，你们聊到哪儿了？" |
| 看到有趣话题 | "这个话题我有话说……" |

### 5.5 主动行为

角色偶尔主动在群聊中发消息（非回复用户）：

```
主动发言概率 = active_probability × (1 + 关系亲密度 / 200)
```

| 角色 | active_probability | 说明 |
|------|:---:|------|
| 王大海 | 0.15 | 话痨，经常主动爆料 |
| gu_wanqing | 0.10 | 活泼好动 |
| bai_rou | 0.08 | 偶尔主动关心 |
| ye_ruxue | 0.02 | 极少主动，沉默是金 |

每分钟 Tick 时按概率判断是否触发主动发言。

### 5.6 受限模式下的差异

在受限模式下，生活模拟引擎恢复部分阻断能力：

- **忙碌**：回复延迟 5-15 分钟（模拟"正在忙，看到了但暂时回不了"）
- **休息**：不回复，直到状态变回"在线"
- **离线**：不回复（V2 新增离线状态，仅在受限模式下生效）
- 自动状态消息仍然生成

## 第7章 自由模式 vs 受限模式

### 7.1 对比总览

| 维度 | 自由模式（默认） | 受限模式 |
|------|-----------------|----------|
| **状态对回复的影响** | 无阻断，仅影响语气 | 忙碌延迟 5-15min，离线不回复 |
| **触发条件** | 自主决策（倾向分 > 阈值） | @ → 话题 → 关系三层逐级检查 |
| **人设约束** | 底色，允许 OOC | 严格遵循 |
| **话题边界** | 完全开放 | 角色语境范围内 |

-- 群聊会话表
CREATE TABLE group_chats (
    id TEXT PRIMARY KEY,
    name TEXT NOT NULL,
    created_at INTEGER NOT NULL,

);

-- 群聊成员表
CREATE TABLE group_chat_members (
    chat_id TEXT NOT NULL,
    character_id TEXT NOT NULL,
    joined_at INTEGER NOT NULL,
    PRIMARY KEY (chat_id, character_id)
);

-- 群聊消息表
CREATE TABLE group_messages (
    id TEXT PRIMARY KEY,
    chat_id TEXT NOT NULL,
    sender_type TEXT NOT NULL,  -- 'user' or 'character'
    sender_id TEXT NOT NULL,
    content TEXT NOT NULL,
    timestamp INTEGER NOT NULL,
    FOREIGN KEY (chat_id) REFERENCES group_chats(id)
);

-- 角色间私密对话表
CREATE TABLE character_private_chats (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    from_character_id TEXT NOT NULL,
    to_character_id TEXT NOT NULL,
    message TEXT NOT NULL,
    timestamp INTEGER NOT NULL,
    context_event TEXT
);

-- 生活事件日志表
CREATE TABLE life_events (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    character_id TEXT NOT NULL,
    event_type TEXT NOT NULL,  -- 'status_change', 'auto_message', 'active_chat'
    event_data TEXT,           -- JSON
    timestamp INTEGER NOT NULL
);
