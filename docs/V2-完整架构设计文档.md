## 第1章 核心设计哲学

### 1.1 事件驱动（Event-Driven）

**所有数值变化必须以事件为唯一驱动源。** AI 不因一句话直接 +5，而是经历一件事后由事件分析器综合判定。

```
用户行为 → 事件(Event) → 事件分析器(Event Analyzer)
  ├── 影响 Relationship 各维度
  ├── 影响 Emotion 实时状态
  ├── 影响 Desire / Boundary / Traits
  ├── 生成 Memory 记录
  └── 可能触发跨角色连锁反应
```

### 1.2 历史解释事件

同一句话在不同历史背景下产生完全不同的事件解释：

| 事件 | 历史上下文 | 效果 |
|------|-----------|------|
| "我爱你" | 长期稳定关系 | Love +2, Trust +1 |
| "我爱你" | 昨天刚出轨 | Love +1, Trust +0, Security -3, Suspicion +5 |

### 1.3 阻尼增长

```
Love 90 → 91：需要 10 倍事件投入
Love 95 → 96：需要纪念日 / 共同旅行 / 长期陪伴
Love 下降：越低越容易继续掉
```

### 1.4 关系阶段化

| 阶段 | 阶段名 | 阈值 | Prompt 变化 |
|------|--------|:---:|------------|
| 1 | 陌生人 | 0-15 | 礼貌疏离 |
| 2 | 认识 | 16-30 | 客气友好 |
| 3 | 朋友 | 31-50 | 自然交流 |
| 4 | 暧昧 | 51-65 | 试探含蓄 |
| 5 | 恋人 | 66-80 | 亲密主动 |
| 6 | 热恋 | 81-90 | 毫不掩饰 |
| 7 | 稳定 | 91-95 | 熟悉默契 |
| 8 | 家人 | 96-100 | 无需多言 |

## 第2章 系统架构

### 2.1 技术栈

| 层级 | 选型 | 理由 |
|------|------|------|
| 前端 | Svelte + Service Worker (PWA) | 体积小、性能好、原生 PWA 支持 |
| 后端 | Python FastAPI + WebSocket | 异步原生、WS 开箱即用 |
| 数据库 | SQLite（服务端） | 零配置、与 V1 兼容、单用户场景够用 |
| LLM | DeepSeek API | 与 V1 一致，成本可控 |
| 配置 | YAML | 人类可读，与 V1 兼容 |

### 2.2 PWA 特性

| 能力 | 实现 |
|------|------|
| 安装到桌面 | Web App Manifest |
| 离线缓存 | Service Worker（缓存静态资源 + 最近对话） |
| 推送通知 | Push API（角色主动消息提醒） |
| 全屏运行 | standalone 模式 |
| 数据持久化 | 全部在服务端 SQLite，PWA 仅做缓存 |

## 第3章 角色数据结构（12 模块）

### 3.1 结构总览

```
Character
├── 1. Core Persona（核心人格，不可变）
├── 2. Relationship（关系，事件驱动变化）
├── 3. Emotion（实时情绪，瞬时变化）
├── 4. Desire（亲密倾向，事件驱动 + 性格约束）
├── 5. Boundary（亲密边界，随关系阶段解锁）
```

## 第4章 事件系统（核心引擎）

### 4.1 Event Log 数据结构

```yaml
event_log:
  event_id: "evt_xxxxx"
  event_type: "conversation"     # conversation / accompany / anniversary / gift / char_interaction / schedule / time_passive / system
  timestamp: "2026-06-26T14:30:00"
  participants:
    - user
    - ye_ruxue
    - bai_rou                   # 可选，群聊/共同事件
  raw_input: "今天辛苦了，早点休息吧"
  
  analysis_result:               # 事件分析器输出
    primary_target: ye_ruxue
    effects:
      - { target: ye_ruxue, engine: relationship, field: trust, delta: +1, weight: 2 }
      - { target: ye_ruxue, engine: emotion, field: touched, delta: +5, weight: 1 }
      - { target: bai_rou, engine: relationship, field: jealousy, delta: +3, trigger: "indirect_observation" }
    
  memory_snapshot: "晚上加班时，用户对夜如雪说了句'辛苦了早点休息'，她表面冷淡但内心微动。"
  
  weight: 2                      # 事件权重（1-100）
```

### 4.2 事件类型与权重

| 事件类型 | 基础权重 | 示例 |
|----------|:---:|------|
| 普通聊天 | 1 | 日常问候、闲聊 |
| 连续聊天（>30min） | 2 | 深入交流 |
| 陪伴（>1h） | 5 | 长时间陪伴 |
| 系统事件（状态变更/节日） | 5 | 生日、状态切换 |
| 纪念日 | 20 | 认识N天、第一次约会日 |
| 共同活动 | 15 | 一起看电影、一起听歌 |
| 送礼 | 30 | 特别的礼物 |
| 第一次表白 | 100 | 关系质的飞跃 |
| 第一次视频/语音 | 80 | 重要里程碑 |
| 共同旅行 | 50 | 深度陪伴 |
| 吵架 | -15 | 负权重事件 |
| 长期冷落 | -8 | 每7天无互动 |

### 4.3 阻尼公式

```
effective_delta = base_delta × event_weight × damping_factor

damping_factor:
  current_value < 50:  1.0
  current_value 50-70: 0.7
  current_value 70-85: 0.4
  current_value 85-95: 0.15
  current_value > 95:  0.05

下降无阻尼：低值时下降全量生效
  current_value < 30:  1.3（加速下跌）
```

### 4.4 跨角色连锁

```
事件：用户陪墨小染看了两小时电影
  ├── 墨小染: Love +3, Security +2, Attachment +3
  ├── 白柔: Security -8, Jealousy +20（通过 Memory 知晓）
  ├── 顾晚晴: Trust -5, Emotion.sad +15（表面微笑）
  └── 夜如雪: 无明显数值变化，但 Memory 记录（"……知道了。"）
```

## 第5章 性癖系统（Fetishes）

### 5.1 三层结构

```yaml
fetishes:
  base:                          # 核心偏好，几乎不变
    - "被温柔触碰颈部"
    - "耳语时的酥麻感"
    - "对方明确的臣服姿态"
    
  acquired:                      # 习得偏好，事件驱动
    - name: "深夜揉肩膀"
      source: "evt_yyy"          # 来源事件
      strength: 45               # 偏好强度（0-100）
      acquired_at: "2026-07-15"
    - name: "被用低沉的语气叫名字"
      source: "evt_zzz"
      strength: 30
      acquired_at: "2026-08-02"
      
  current_active: []             # 运行时计算：mood=happy 全部激活，mood=sad 收缩
```

### 5.2 动态规则

| 触发条件 | 效果 |
|----------|------|
| 极度幸福事件 | 可能解锁新的 acquired_fetish |
| 长期未触发某偏好 | acquired 中强度降低 |
| 情绪 sad/angry | current_active 收缩到 base 安全区 |
| 情绪 happy/excited | current_active 扩大，临时激活隐藏偏好 |
| 关系阶段提升 | 解锁新的 base 区域偏好 |

### 5.3 各角色差异化示例

| 角色 | base 特征 | acquired 方向 |
|------|----------|--------------|
| 夜如雪 | 冷淡主导型，偏好少而精 | 长期相处后悄悄多出"被揉肩膀""被记住她的习惯" |
| 花梨 | 纯被动型，偏好极简 | 第一次被抱在怀里看星星，acquired 多出"从背后被轻轻抱住" |
| 林糖糖 | 主动撩拨型，偏好丰富 | 被反撩后 acquired 多出"被看穿伪装""被对方抢先一步" |
| 墨小染 | 占有标记型 | acquired 随安全感提升从"束缚"转向"信任中的亲密" |

## 第6章 对话系统

### 6.1 消息格式

```json
{
  "message_id": "msg_xxx",
  "sender_type": "character",
  "sender_id": "ye_ruxue",
  "display_name": "夜如雪",
  "avatar_url": "/static/avatars/ye_ruxue.png",
  "content": "……说重点。",
  "action": "夜如雪头也不抬地翻着案卷，指尖在桌面上轻轻敲了两下",
  "inner_thought": "（其实她注意到你今天穿了新衬衫，只是懒得说）",
  "mood_tag": "冷静",
  "timestamp": "2026-06-26T14:30:01"
}
```

### 6.2 动作描述机制

每个角色回复可附带 `action`（动作描述）和 `inner_thought`（内心独白），由 LLM 在生成回复时同时输出。

**动作描述格式**：
```
[动作]夜如雪轻轻放下钢笔，抬眼看了你一眼[/动作]
……说重点。
[内心]这家伙今天倒是来得早。[/内心]
```

LLM 的 System Prompt 中注入生成指令：
```
你的回复格式为：
[动作]角色动作或表情描述[/动作]
对话内容
[内心]角色没有说出口的真实想法[/内心]

要求：
- 动作描述简洁，1-2句
- 内心独白体现角色的真实想法，可能与表面言行相反
- 内心独白中不要包含过多解释，要像真的在心里想
```

### 6.3 私聊 vs 群聊差异

| 维度 | 私聊 | 群聊 |
|------|------|------|
| 回复长度 | 正常 | 偏短 |
| 内心独白频率 | 高 | 极低（群聊不展示内心独白） |
| 动作描述 | 细腻 | 简洁 |
| 话题边界 | 无限制 | 受角色 chat_tags 影响 |
| 性格差异 | 按角色自然展开 | 保持群聊社交面具 |

### 6.4 表情/图片/语音（未来）

```
V2.2 计划：
- 角色内置表情集（适合微信风格的 emoji+自定义表情）
- 语音消息：TTS 文字转语音（每个角色独立声线）
- 图片消息：角色可发送"生活照片"（AI 生成图片）
```

预留字段：`content_type: text | action | emoji | voice | image`

## 第7章 群聊系统

### 7.1 群聊类型

| 类型 | 说明 | 创建方式 |
|------|------|----------|
| 大厅 | 默认群聊，所有 11 个角色都在 | 系统自动 |

### 7.2 成员选择 UI

```
创建新群聊：

│  选择成员（至少选2人）              │
│                                   │

│  ☐ 柳青柠  ●在线                  │

│                                   │

└──────────────────────────────────┘
```

### 7.3 动态加入/退出

- 用户可在群聊设置中随时增减成员
- 成员加入时：广播系统消息 "夜如雪加入了群聊"
- 成员退出时：该角色的对话历史仍保留在群聊记录中
- 群聊可归档（不再活跃但不删除）

### 7.4 角色间互动（群聊内）

```
触发条件：
- @另一个角色 → 角色间对话概率 +40
- 话题涉及另一角色的 chat_tags → +20
- 关系亲密且同在一个群 → 偶尔互相引用

示例：
王大海：如雪姐最近那个案子怎么样了？
```

## 第11章 关键技术决策

| 决策 | 理由 |
|------|------|
| 事件是唯一数值变化入口 | 审计链完整，历史可回溯 |
| 阻尼增长公式 | 避免数值"通货膨胀"，高值极难增长 |
| base/current 双层结构 | 角色核心永远不变，但表面有成长 |
| 关系阶段化 Prompt | 不同阶段语言风格完全不同 |
| PWA 而非原生 App | 一套代码全平台，开发成本最低 |
| 全部数据服务端持久化 | 不丢记录，PWA 仅做缓存加速 |
| 每个角色独立 LLM 上下文 | 互不污染，角色性格独立 |
